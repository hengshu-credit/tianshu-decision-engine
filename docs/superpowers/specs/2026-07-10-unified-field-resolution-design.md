# 天枢引擎统一字段解析与测试执行链路设计

日期：2026-07-10
状态：已确认，按方案 B 实施

## 1. 背景

当前规则设计、字段分析、测试参数生成、脚本编译、运行时参数绑定和结果展示分别存在多套实现。同一个引用在不同页面和执行阶段可能被解释成字段、字面量或输入参数，导致以下已确认问题：

1. 规则详情测试接口返回 `result`，页面却按模型测试的 `outputs` 读取，部分规则的输出结果不展示；`0`、`false`、空对象等合法结果也容易被真值判断隐藏。
2. JCZR 基础准入的 `age` 测试参数为字符串 `"22"`，执行边界未按字段类型绑定，QLExpress 在字符串和整数之间执行 `<` 时失败。
3. JCLTest 决策流中的函数调用参数 `DAY` 被字段分析器当成输入字段，而编译器又将它当作未加引号的变量；设计器测试参数生成没有递归合并嵌套规则 JCZR 所需的 `score_f1_fields`。
4. 设计器中的字段槽位和值槽位缺少统一的操作数语义。函数参数等值槽位中的手输内容，本应作为字面量，却可能因内容形似标识符而被关联为字段。
5. 规则详情和模型详情各自构造字段展示信息，数据对象字段仅显示末级字段名，丢失对象路径。

这些问题不是单点页面缺陷，而是同一份规则元数据在多套解析逻辑之间发生语义漂移。因此采用方案 B：保留现有模型 JSON 和公开接口的兼容性，新增统一的引用语义、依赖解析计划、测试结构生成和执行参数绑定能力，再通过薄适配层供各页面和执行入口复用。

## 2. 目标

### 2.1 功能目标

- 规则详情和模型详情始终展示测试输出，包括对象、数组、字符串、数字、布尔值和 `null`。
- 测试 JSON、规则输入输出字段、项目接口文档和运行时绑定使用同一份字段解析结果。
- 九类规则设计器以相同语义处理条件左右值、动作目标、动作值和函数参数。
- 嵌套规则递归合并真正的外部输入，并能读取父规则刚计算出的中间变量。
- API、数据库、名单、模型和计算变量的依赖只展开到真正的叶子输入。
- 数据对象字段使用完整对象路径和完整展示名称。
- 新配置显式保存引用语义，旧配置通过兼容适配规则得到确定且一致的解释。

### 2.2 统一依赖原则

引擎中的依赖节点分为以下三层：

| 层级 | 类型 | 是否由调用方直接输入 | 解析方式 |
|---|---|---:|---|
| 叶子层 | 输入变量 `INPUT` | 是 | 按声明类型绑定 |
| 叶子层 | 数据对象 `DATA_OBJECT` | 是 | 按完整对象路径生成嵌套 JSON；对象自身依赖继续展开 |
| 中间层 | 计算变量 `COMPUTED` | 否 | 在规则执行过程中产生，可作为下游依赖 |
| 派生层 | 接口变量 `API` | 否 | 递归解析其输入、数据对象或计算变量依赖 |
| 派生层 | 数据库变量 `DB` | 否 | 递归解析其输入、数据对象或计算变量依赖 |
| 派生层 | 名单变量 `LIST` | 否 | 递归解析其输入、数据对象或计算变量依赖 |
| 派生层 | 模型 `MODEL` | 否 | 递归解析模型输入依赖；模型输出作为可引用中间结果 |

常量不属于外部输入。函数自身不是输入节点，函数参数中的引用才参与依赖解析，字面量不参与依赖解析。

### 2.3 非目标

- 不重写九类设计器的模型格式。
- 不引入新的规则 DSL。
- 不自动修改用户输入的变量编码、模型编码、对象字段名或大小写。
- 不以名称或编码替代 ID 持久化关联；编码匹配只用于手输字段槽位的即时解析和旧数据兼容，解析成功后仍保存 ID。
- 不改变现有规则同步、发布和客户端调用的公开契约；新增字段应保持旧客户端可忽略。

## 3. 统一语义模型

### 3.1 引用描述 `ResolvedReference`

统一解析后的引用至少包含：

```text
refId              稳定实体 ID
refType            VARIABLE / MODEL / DATA_OBJECT_FIELD / CONSTANT / RULE
code               用户原样保存的编码或完整路径
label              完整展示名称
valueType          STRING / INTEGER / LONG / DECIMAL / BOOLEAN / DATE / DATETIME / ARRAY / OBJECT
sourceType         INPUT / DATA_OBJECT / COMPUTED / API / DB / LIST / MODEL / CONSTANT
scriptName         编译时脚本名
dependencies       直接依赖的引用 ID
objectId           数据对象 ID，可空
objectCode         数据对象编码，可空
objectLabel        数据对象名称，可空
fieldPath          对象内字段路径，可空
```

引用关联以 `refId + refType` 为准。`code` 用于脚本和展示，不作为已持久化关系的主键。

### 3.2 操作数 `Operand`

条件、动作和函数参数统一转换为操作数：

```text
kind       REF / LITERAL / EXPRESSION
value      原始内容，禁止自动改变用户编码
refId      REF 时必填
refType    REF 时必填
valueType  已知时填写
```

本期设计器只主动产生 `REF` 和 `LITERAL`。`EXPRESSION` 保留给脚本编辑器和已有明确表达式配置，不通过文本形态猜测产生。

### 3.3 字段槽位与值槽位

所有设计器控件按语义分为两类：

1. 字段槽位：条件左值、赋值目标、结果变量等。通过选择器选择时为 `REF`；手输内容与当前可用引用的 `code` 精确匹配时解析为 `REF`，并在保存时持久化 `_varId/_refType`。无法解析时阻止保存并提示“字段编码不存在”，不静默生成脚本变量。
2. 值槽位：条件阈值、赋值内容、函数参数、选项值等。通过选择器选择时为 `REF`；手输内容一律为 `LITERAL`，即使内容恰好与某个字段编码相同。需要引用字段时必须使用字段选择器。

该规则解决“手输 code 关联字段”和“函数参数手输选项直接使用”之间的歧义。

### 3.4 旧模型 JSON 兼容规则

- 存在 `_varId/_refType` 或函数参数对应 `_argRefs[index]` 时，按 `REF` 解析。
- 函数参数对应 `_argRefs[index]` 为 `null` 或不存在时，按 `LITERAL` 解析；例如 `DAY` 编译为 `"DAY"`。
- 旧条件或动作的字段槽位缺少 ID 时，允许通过项目作用域内的完整 `code` 精确匹配一次；唯一匹配后在下次保存时回填 ID。
- 出现同编码多实体、跨项目实体或无法唯一匹配时，不猜测，返回可定位到节点和槽位的校验错误。
- 旧脚本编辑器内容保持脚本语义，继续通过 `scriptVarRefs` 保存显式引用映射。

## 4. 后端统一解析架构

### 4.1 `FieldDependencyResolver`

服务端新增统一字段依赖解析服务，输入为项目、规则/模型/变量入口和可选的未保存 `modelJson`，输出不可变的 `ResolutionPlan`。解析过程：

1. 从结构化模型中提取显式引用和操作数。
2. 以 `refId + refType` 加载实体，不通过名称回溯已持久化引用。
3. 深度优先解析变量来源、模型输入、数据对象依赖和嵌套规则。
4. 对每个节点记录访问状态，发现环时返回完整环路，例如 `规则A -> 变量B -> 模型C -> 规则A`。
5. 对结果稳定排序并去重：先按依赖拓扑顺序，再按首次出现顺序，保证各页面生成一致 JSON。
6. 将外部输入、内部计算节点、运行时派生节点和输出字段分别保存在计划中。

`ResolutionPlan` 至少提供：

```text
externalInputs      仅 INPUT 和 DATA_OBJECT 叶子
runtimeNodes        COMPUTED / API / DB / LIST / MODEL
outputs             规则或模型输出
references          全量引用索引
dependencyEdges     血缘和诊断使用
diagnostics         兼容回填、缺失引用、循环等信息
```

### 4.2 现有解析器整合

- `RuleFieldAnalyzer` 改为 `ResolutionPlan` 的持久化适配器，不再自行递归推断。
- `RuleModelVarParser` 只负责将九类模型 JSON 标准化为显式操作数和直接引用，不再决定某段文本是否是输入字段。
- `VariableSourceResolver` 复用统一依赖遍历和节点分类，不再维护另一套叶子变量规则。
- 项目接口说明、血缘分析、规则/模型输入输出字段、规则同步和测试参数结构均消费同一计划。
- 对未保存的设计器内容，解析服务接收当前 `modelType + modelJson`，不要求先写数据库。

### 4.3 统一测试结构接口

新增测试结构请求，支持规则、模型和变量三类入口：

```text
POST /api/rule/test-schema
request:  projectId, targetType, targetId?, modelType?, modelJson?
response: inputs, outputs, sampleParams, diagnostics
```

该接口由独立的统一测试结构控制器提供；现有规则、模型和变量控制器可在兼容期保留原接口，但内部必须委托同一服务。`sampleParams` 由后端按字段类型生成，前端只负责展示和允许用户编辑。

嵌套对象生成规则：

- `bankcard.bank_card_no` 生成 `{ "bankcard": { "bank_card_no": "" } }`。
- 模型运行时对象如 `score_f1_fields` 由模型依赖定义生成，不添加模型输出 `score_f1.score`。
- 同一对象的多个字段合并到同一结构。
- 父子路径冲突时返回诊断错误，不用后写值覆盖先写对象。
- 计算变量、函数选项和常量不出现在输入 JSON。

### 4.4 执行参数绑定 `ExecutionParameterBinder`

所有规则测试、模型测试、变量测试、项目执行和实验执行在进入 QLExpress 前使用同一绑定器。绑定器依据 `ResolutionPlan.externalInputs` 或同步下发的字段元数据完成：

- 整数类型：字符串 `"22"` 转为 `Integer/Long`，超界或非法内容返回字段级错误。
- 小数类型：字符串转为声明的小数类型，避免二进制精度猜测。
- 布尔类型：只接受布尔值及明确配置的字符串表示。
- 数组和对象：验证容器形态并递归绑定已声明子字段。
- 日期和时间：按字段声明格式规范化；未声明格式时保留现有 ISO/字符串兼容行为。
- 字符串：保留用户原值，不因内容形似数字自动转换。
- `null`：依据字段是否允许为空处理，不转换成字符串 `"null"`。

绑定失败统一返回字段完整路径、期望类型、实际值类型和安全摘要。JCZR 的 `age: "22"` 将被绑定为数值后再执行比较。

## 5. 编译与运行时上下文

### 5.1 操作数编译

所有编译器调用统一操作数编译器：

- `REF`：通过 `VarContext` 的 ID 映射获得脚本名。
- `LITERAL`：按声明或推导类型安全转义；字符串加引号，数字和布尔值保持原生字面量，`null` 使用空值。
- `EXPRESSION`：仅来自显式脚本/表达式配置，原样进入受支持的编译路径。

函数调用 `idCardAge(idcard_no, credit_time, DAY)` 的旧配置中，前两个 `_argRefs` 有引用，第三个为 `null`，应编译为：

```text
idCardAge(idcard_no, credit_time, "DAY")
```

而不是将 `DAY` 作为输入字段或未声明脚本变量。

### 5.2 计算结果同步

父规则的计算变量必须可被后续派生变量和嵌套规则读取。动作编译器在赋值、函数返回值写入以及其他产生目标变量的动作后，统一调用运行时上下文写入函数：

```text
age = idCardAge(idcard_no, credit_time, "DAY");
setRuntimeValue("age", age);
```

`setRuntimeValue`：

- 写入当前规则执行帧的共享上下文。
- 支持完整对象路径，使用统一的嵌套路径写入器。
- 不覆盖无关字段。
- 在嵌套规则返回后保持父规则上下文可继续使用。
- 对并发请求使用独立执行帧，不共享可变全局状态。

嵌套 `executeRule` 从当前执行帧构造子规则输入，并按子规则的 `ResolutionPlan` 读取所需字段。这样 JCLTest 先计算 `age`，再调用 JCZR 时，JCZR 能读取数值年龄，同时其模型依赖继续展开为 `score_f1_fields`。

### 5.3 错误传播

- 嵌套执行失败时保留子规则编码、节点路径和原始根因，不只返回空异常。
- 编译期缺失引用、运行期缺失必填输入、类型绑定失败和依赖循环使用不同错误码。
- 页面测试弹窗展示简洁消息，同时保留可展开的详细执行链路。

## 6. 前端统一适配

### 6.1 引用目录

新增共享引用目录构建器，供 `varPickerMixin`、规则详情、模型详情、测试页面和九类设计器复用。展示规则：

- 普通变量：`年龄 age`
- 数据对象字段：`银行卡信息/银行卡号 bankcard.bank_card_no`
- 模型输出：`反欺诈评分F1/评分 score_f1.score`
- 常量：`常量名称 constant_code`

目录保留 `refId/refType/sourceType/valueType`，选择后控件同时写入显示内容和稳定 ID。

### 6.2 设计器控件

条件编辑器、动作编辑器、函数参数编辑器和高级模型维度编辑器复用统一的 `OperandEditor` 语义层；不要求一次重写全部页面外观。现有 Element UI 控件通过 mixin/工具函数接入：

- 字段槽位允许选择或手输 code，失焦/保存时解析。
- 值槽位允许选择引用或手输字面量。
- 选择器结果写入显式引用元数据。
- 从选择模式改为手输时清除旧引用 ID，避免“显示值已改、底层仍引用旧字段”。
- 从手输改为选择时覆盖字面量语义。

### 6.3 测试参数生成

- 九类设计器和规则详情不再各自遍历模型 JSON 猜测输入字段。
- 保存中的规则详情按目标 ID请求测试结构。
- 未保存的设计器按当前 `modelJson` 请求测试结构。
- 前端保留纯函数后备适配器，供离线单元测试和后端不可用时给出明确错误；该适配器消费标准化引用，不另建依赖规则。
- `testSampleParams.js` 只负责根据统一字段描述构造值，不负责发现字段依赖。

### 6.4 测试结果展示

新增共享结果归一化函数：

```text
规则响应 result   -> output
模型响应 outputs  -> output
变量响应 value    -> output
```

展示判断以“响应是否包含结果字段”为准，不使用结果值真值判断。以下值全部可见：`0`、`false`、`""`、`{}`、`[]` 和 `null`。规则详情、模型详情、统一测试弹窗和独立测试页复用同一格式化组件或工具函数。

## 8. 数据流

### 8.1 保存和字段分析

```text
设计器模型 JSON
  -> 操作数标准化
  -> FieldDependencyResolver
  -> ResolutionPlan
  -> 持久化输入/输出字段 + 血缘边
```

### 8.2 测试结构

```text
规则/模型/变量 ID 或未保存模型 JSON
  -> FieldDependencyResolver
  -> externalInputs
  -> 嵌套路径构造
  -> sampleParams
```

### 8.3 执行

```text
用户 JSON
  -> ExecutionParameterBinder
  -> 变量/模型来源求值
  -> QLExpress
  -> setRuntimeValue
  -> 嵌套规则/后续节点
  -> 统一结果响应
```

## 9. 实施顺序

1. 先为已报告问题补充失败测试，锁定现状。
2. 实现操作数标准化和函数参数字面量编译，修正 `DAY`。
3. 实现统一依赖计划，接管规则字段分析和测试结构生成，修正 JCLTest 输入。
4. 实现执行参数绑定，修正 JCZR 字符串数字比较。
5. 实现运行时计算结果同步和嵌套规则上下文，贯通 JCLTest -> JCZR。
6. 实现共享引用目录、完整数据对象展示和统一结果归一化。
7. 将九类设计器、规则/模型/变量测试、项目接口和实验逐个迁移到统一逻辑。
8. 删除迁移后不再被使用的重复递归和启发式逻辑；不做无关重构。
9. 执行单元、集成、构建、启动和浏览器全链路验证。

每一步都先运行目标失败测试，再做最小实现，并在通过后运行受影响模块回归。

## 10. 测试设计

### 10.1 后端单元测试

- 九类模型的引用提取及操作数标准化。
- 条件左/右值分别覆盖选择引用、手输字段 code、手输字面量。
- 动作赋值目标和值、函数参数、函数返回目标覆盖引用和字面量。
- `_argRefs` 有引用、为 `null`、长度不足及旧模型缺失字段。
- INPUT、DATA_OBJECT、COMPUTED、API、DB、LIST、MODEL 的单层和多层依赖。
- 数据对象自身依赖输入字段。
- 嵌套规则、重复依赖去重、缺失 ID、跨项目引用和依赖循环。

## 11. 完成标准

以下条件必须同时满足才可声明完成：

- 五个已报告问题均有可重复的回归测试并通过。
- 所有测试参数来源于统一依赖计划，页面不再维护独立的字段发现规则。
- 运行时输入类型在统一边界完成绑定。
- 父规则计算变量能稳定传给嵌套规则和后续派生节点。
- 所有新引用按 ID 持久化，旧配置有确定的兼容行为。
- 九类设计器和覆盖矩阵中的功能均有自动化测试或真实浏览器验证证据。
- 前后端完整质量门禁通过，没有通过弱化断言、跳过测试或隐藏错误取得“假通过”。

## 12. 风险与控制

- **兼容风险**：旧 JSON 缺少引用元数据。通过槽位语义和唯一 code 匹配适配，并用诊断提示不可判定配置。
- **递归风险**：规则、变量和模型可能形成环。统一解析器在执行前检测并报告完整环路。
- **范围风险**：消费者较多。按解析核心、执行核心、前端适配、全链路验收分层迁移，每层保持现有公开契约。
- **运行时风险**：上下文同步涉及嵌套调用。使用请求级执行帧和路径写入测试覆盖并发隔离。
- **展示风险**：不同接口结果字段不同。只在共享归一化层兼容接口差异，页面统一消费 `output`。

该设计以“一份引用语义、一份依赖计划、一个参数绑定边界、一个结果展示模型”为约束，新增场景必须扩展统一层并补充覆盖矩阵，禁止在单个页面或编译器中再次新增平行解析规则。
