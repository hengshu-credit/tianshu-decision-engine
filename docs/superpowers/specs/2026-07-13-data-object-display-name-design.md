# 数据对象显示名称调整设计

## 目标

引擎前端所有数据对象字段引用统一展示为：

```text
数据对象名称/字段名称 完整字段路径
```

示例：

```text
银行卡信息/银行卡号 bankcard.bank_card_no
```

普通变量、常量、模型输出和函数的展示格式保持不变。

## 实现范围

1. 修改前端公共引用目录 `referenceCatalog.js`：
   - 数据对象字段的语义标签由“字段名称”改为“数据对象名称/字段名称”。
   - `refCode` 继续使用数据对象脚本路径与字段脚本路径组成的完整字段路径。
   - `displayName` 继续由语义标签与完整路径拼接。
2. 修改模型详情的备用引用映射：
   - 当公共引用目录状态不可用时，仍按同一格式构造 `varLabel` 和 `varLabelText`。
3. 不修改后端接口、实体或数据库结构，不改变任何引用 ID。
4. 更新公共引用目录、模型详情、规则详情和设计器中的相关行为测试。

## 数据流

```text
数据对象名称 + 字段名称 + 对象脚本路径 + 字段脚本路径
  -> 公共引用目录 refLabel/refCode
  -> 选择器、详情页、公式与追踪展示
```

已保存的引用仍通过 `refId + refType` 解析。页面重新加载时会用当前元数据生成最新显示名称，不依赖业务名称进行关联。

## 测试与验收

1. 先修改测试期望并确认旧实现失败。
2. 修改最小生产代码使定向测试通过。
3. 运行前端全量测试。
4. 启动前后端，在真实模型详情和规则/设计器引用选择中检查展示格式。
5. 检查浏览器控制台无错误。

## 数据库导出与 Git

1. 功能验证完成后，使用当前本地 MySQL 容器导出 `rule_engine` 数据库。
2. 导出文件保存到 `rule-engine-server/src/main/resources/sql/`，沿用现有 `export_YYYYMMDDHHmm.sql` 命名方式。
3. 检查导出文件包含表结构和数据，且不包含命令报错文本。
4. 将代码、测试、规格和数据库导出一起提交到本地 `master`，再推送远程 `master`。

## 非目标

- 不修改数据对象、字段的持久化名称。
- 不改变脚本路径生成规则。
- 不为历史导出文件做批量改写。
- 不引入新的前端或后端依赖。
