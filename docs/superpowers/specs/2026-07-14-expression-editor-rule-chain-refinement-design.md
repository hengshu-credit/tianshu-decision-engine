# 表达式编辑器与额度规则链优化设计

## 目标

在现有统一表达式 AST 和服务端执行能力上，解决复杂表达式难以查找资源、树过深难以阅读的问题，并把额度公式依赖的月成功还款额、风险系数、风险额度拆成可独立维护的规则，最终由额度决策流顺序执行并引用其结果。

## 左侧双栏资源选择器

`ExpressionPalette` 改为与 `VarPicker` 一致的双栏结构：左侧为固定宽度分类，右侧只展示当前分类内容。分类包含手动输入、普通变量、常量、数据对象、模型、函数/方法、名单查询、运算符、取值与转换；每项显示不随搜索变化的总数，分类文字和数量保持单行。

右栏顶部提供搜索框，字段和函数列表按每页 50 项分页。搜索只影响当前分类的右栏结果，不改变左侧分类宽度和总数。项目已有的 `_ref.category`、`_varId`、`_refType` 和函数稳定 ID 继续作为分类、定位和插入依据。

手动输入分类提供“输入阈值”和“输入字段路径”两种入口。字段路径输入后复用字段选择器的解析语义：唯一编码或对象路径匹配时生成带 `refId`、`refType` 的已解析 `PATH`，等价绑定到现有字段；出现多个同编码候选时展示候选列表，用户明确选择后再插入；无匹配时才保留未解析 `PATH`。因此手输路径不会退化为名称关联。

## 中间表达式折叠

折叠是纯前端视图状态，不写入 Operand AST。每个包含子节点的表达式节点显示独立的展开/折叠按钮；折叠后保留节点摘要和隐藏子节点数量。工作区提供“展开全部”和“折叠到两层”操作。

复杂表达式首次打开时默认折叠深度大于等于两层的子树。用户选择、插入或更新深层节点时，编辑器自动展开该路径的所有祖先，保证当前编辑位置始终可见。表达式发生结构变更后，失效路径从折叠集合中清理。折叠状态不进入撤销/重做历史，也不影响公式预览、校验、保存和服务端执行。

## 全局字段复用

直接复用已有全局字段：

- `credit_limit`：授信额度
- `available_credit_limit`：可用额度
- `used_credit_limit`：已使用额度
- `age`：年龄
- `score`：反欺诈分 F1/score
- `amount`：最终额度

只新增三个没有等价全局字段的 NUMBER/COMPUTED 变量：

- `monthly_success_repayment_amount`：月成功还款额
- `risk_factor`：风险系数
- `risk_amount`：风险额度

此前验收创建的项目级 `monthlyRepayment`、`usedAmount`、`riskFactor`、`riskAmount`、`amount` 在规则引用迁移完成后通过前端删除，避免项目字段遮蔽全局字段。

## 三张复杂交叉表规则

三个规则均为 GLOBAL / `CROSS_ADV`，结果字段通过稳定变量 ID 持久化：

1. `MONTHLY_REPAYMENT_MATRIX`：行维度 `credit_limit`，列维度 `available_credit_limit`，输出 `monthly_success_repayment_amount`。录入需求表中的 4 × 11 区间和全部 44 个单元格。
2. `RISK_FACTOR_MATRIX`：行维度 `age`，列维度 `score`，输出 `risk_factor`。录入 4 × 4 风险系数矩阵。
3. `RISK_AMOUNT_MATRIX`：行维度 `age`，列维度 `score`，输出 `risk_amount`。录入 4 × 4 风险额度矩阵。

区间统一使用左闭右开语义；负无穷和正无穷使用设计器现有范围表达方式。每条规则保存并编译后通过规则测试页面分别验证边界值与代表值。

## 额度决策流

项目规则 `AMOUNT_CALC` 调整为四个串行执行动作节点：

1. 调用 `MONTHLY_REPAYMENT_MATRIX`。
2. 调用 `RISK_FACTOR_MATRIX`。
3. 调用 `RISK_AMOUNT_MATRIX`。
4. 将下列表达式结果赋给全局 `amount`：

```text
CEIL(max(4200,
         min((min(max(monthly_success_repayment_amount, used_credit_limit), 9000)
              * risk_factor * 0.3
              + risk_amount * 0.5),
             7000)) / 500) * 500
```

规则调用继续使用现有 `RULE_CALL` 动作和运行时上下文，子规则输出保留在同一执行上下文中，最终表达式通过变量 ID 引用三个计算结果。

## 异常与交互提示

- 当前分类无结果时显示明确空状态，不切换到其他分类。
- 路径解析多候选时禁止直接插入并提示选择具体字段。
- 折叠节点仍可被选中和删除，节点摘要始终展示完整公式片段的省略视图。
- 规则调用、矩阵单元或表达式参数未配置完整时沿用现有校验并阻止保存或应用。
- 左侧提供一句简短提示，说明“先选中中间位置，再点击右侧资源；路径会优先反解为已有字段”。

## 验收

- 单元测试覆盖资源分类、分类计数、搜索分页、路径唯一/多候选/无匹配、节点折叠、批量展开、选择路径自动展开和 AST 不变性。
- 前端全量 Jest、后端 `mvn clean install -DskipTests` 与 `mvn test` 全部通过。
- 后端和前端分别启动，浏览器逐项验证双栏资源区、手输路径反解、复杂公式折叠、三张规则配置与测试、决策流规则链和最终额度结果。
- 浏览器控制台不得出现 `aria-hidden`、重复 key 或运行错误。
- 完成后停止服务，将 `codex/unified-expression` 合并到 `master`，在 `master` 再执行必要的合并后验证。
