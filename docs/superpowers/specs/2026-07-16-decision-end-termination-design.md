# 决策树与决策流结束节点终止语义设计

## 目标

统一决策树和决策流的结束节点语义，使结束节点成为用户主动选择的控制流节点，而不是模型合法性的必选占位符。决策树和决策流初始化时只创建开始节点；决策流在没有结束节点时也可以保存、编译和发布，并在没有后继节点时自然结束。

结束节点提供两种明确的终止范围：跳出当前规则和跳出整体规则。跳出当前规则只结束当前脚本执行；当前规则作为子规则时，调用它的外层规则继续执行。跳出整体规则会穿透任意层级的子规则调用栈，根规则后续节点和后续规则均不再执行。

## 模型契约

继续使用现有 `end-event` 前端节点类型和后端 `type=end`，不新增第二套节点类型。结束节点在 `properties` 及后端 `nodes` 中持久化 `terminationScope`：

- `CURRENT_RULE`：跳出当前规则。
- `ALL_RULES`：跳出整体规则。

历史模型中的结束节点没有 `terminationScope` 时统一按 `CURRENT_RULE` 解释，保持历史节点只终止当前规则路径的兼容语义。加载历史模型后再次保存时写入显式范围。结束节点仍禁止作为连线起点。

## 初始化与前端交互

共享初始化模板 `getDefaultFlowData()` 只生成一个开始节点，决策树和决策流均不再默认放置结束节点。

用户在任一设计器点击“结束”后不立即创建节点，而是打开共享的结束节点确认窗口。窗口内必须先选择终止范围，并展示对应影响：

- 跳出当前规则：执行到此立即返回当前规则结果；作为子规则时，外层规则继续执行。
- 跳出整体规则：执行到此立即终止根规则；即使当前规则是子规则，外层及其后续所有规则也不再执行。

默认选中风险较低的 `CURRENT_RULE`。用户确认后才创建节点，取消则不修改画布。节点使用颜色和内部文字同时区分语义：

- `CURRENT_RULE`：橙色，节点文字为“返回”。
- `ALL_RULES`：红色，节点文字为“终止”。

属性面板和节点类型说明同步显示“结束当前规则”或“结束整体规则”，避免只依赖颜色辨识。序列化、反序列化、旧模型转换和 LogicFlow 渲染均保留 `terminationScope`。

## 编译语义

`DecisionFlowCompiler` 删除“至少存在一个 end 节点”的结构限制，继续保留开始节点唯一、节点和连线引用有效、禁止有向环等校验。`DecisionTreeCompiler` 维持现有不强制结束节点的行为。

图编译前先收集当前规则输出变量，再由 `GraphScriptGenerator` 按结束节点范围生成控制语句：

- `CURRENT_RULE`：构造当前规则输出 Map，并生成 QLExpress `return`。已赋值输出返回当前值，未赋值输出保持现有的 `null` 初始化语义；没有输出字段时返回 `null`。
- `ALL_RULES`：生成对受控运行时函数 `terminateAllRules()` 的调用。该函数不作为业务异常处理，而是发出专用整体终止信号。

没有结束节点的路径继续自然执行到脚本末尾，并使用现有 `RuleScriptResultCollector` 汇总输出。这样结束节点是显式提前返回，而不是正常完成规则所必需的节点。

## 整体终止运行时

在 core 中定义专用 `RuleTerminationSignal`，只表示受控的整体规则终止。服务端 `RuleRuntimeInvoker` 和客户端 `ClientRuleRuntimeInvoker` 都向 QLExpress 注册 `terminateAllRules()`，由该函数根据根执行帧构造结果并抛出信号。

根执行帧在开始执行时记录根规则输出脚本名。整体终止发生时，从当前共享执行上下文按这些脚本名读取值并构造根规则结果 Map：已赋值字段保留当前值，尚未赋值字段写入 `null`，键顺序遵循根规则输出字段顺序。嵌套路径按现有脚本字段路径读取。

`QLExpressEngine` 识别 QLExpress 可能包装过的 `RuleTerminationSignal` 并继续抛出，不将其转换为失败的 `RuleResult`。信号经过所有子规则运行器继续向根边界传播；子规则调用点后续表达式不再执行。服务端 `RuleExecuteService` 和客户端 `RuleEngineClient` 只在根执行边界捕获信号，并生成 `success=true`、`result=根规则输出Map` 的正常结果，执行日志也按成功的受控终止记录。

普通运行时异常仍按现有失败流程处理，不能被误判为整体终止。整体终止函数在没有根执行帧时调用属于非法运行状态并返回执行失败。

## 同步与兼容

为保证服务端执行和客户端 SDK 本地执行一致，已发布规则同步数据增加非数据库字段 `outputScriptNames`：

- `RulePublished` 使用非表字段承载输出脚本名。
- `RulePushMessage` 在发布推送时携带输出脚本名。
- `CachedRule` 缓存输出脚本名。
- HTTP 单条同步、全量同步和 Redis 发布更新均复制该字段。

该变更不增加数据库列。旧客户端会忽略新增 JSON 字段；新客户端读取没有该字段的旧缓存时按空输出集合处理。新结束节点必须由新版本服务端编译发布，因此正式使用整体终止时，服务端和客户端 SDK 应同步升级。

## 测试与验收

后端自动化测试覆盖：

- 决策流无结束节点可以编译，决策树现有无结束节点行为保持不变。
- 当前规则结束节点生成并执行 `return`，后续节点不执行，输出 Map 与未赋值 `null` 语义正确。
- 整体结束节点生成终止调用，普通异常不会被识别成终止信号。
- 服务端多层子规则中触发整体终止后，父规则后续逻辑不执行，最终返回根规则输出契约。
- 客户端 SDK 本地多层子规则具备相同行为。
- 单条同步、全量同步和 Redis 推送均保留 `outputScriptNames`。
- 历史无 `terminationScope` 的结束节点按 `CURRENT_RULE` 编译。

前端自动化测试覆盖：

- 默认模板只包含开始节点。
- 点击结束按钮后必须经过范围选择和确认，取消不创建节点。
- 两种范围写入正确的 `terminationScope`，具有不同颜色和文字。
- 模型保存、加载和旧格式转换不丢失终止范围。
- 决策流验证不再要求结束节点。

完成实现后执行仓库要求的完整校验：后端 `mvn clean install -DskipTests`、启动 `rule-engine-server`、`mvn test`；前端启动开发服务确认页面无错误、停止服务后执行 `npm test` 和 eslint 检查。最后通过浏览器分别在决策树和决策流中验证初始化、结束节点选择弹窗、两种颜色、保存编译发布、当前规则返回和嵌套规则整体终止流程。
