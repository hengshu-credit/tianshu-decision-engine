# 统一操作数选择器设计

## 目标

将规则设计器、模型字段关联、模型字段默认值和分流实验条件分流中的执行语义输入统一为同一套操作数选择方式。用户点击字段或值位置后打开统一选择面板，再选择手输阈值、手输路径、变量、常量、数据对象、模型或函数。废弃输入框后的手动编码切换按钮、条件树中的常量/变量下拉和由切换动作触发的自动弹层。

项目尚未正式发布，本次直接升级当前数据结构、示例数据和测试夹具，不保留旧交互或旧 modelJson 的兼容分支。

## 范围

本次覆盖：

- 九种规则设计器中参与规则执行的字段、条件值、动作值和结果目标。
- 决策树、决策流、决策表、规则集和分流实验共用的条件树。
- 动作块中的赋值目标、赋值值、条件值、函数参数和其他参与表达式生成的操作数。
- 模型输入字段来源关联、模型输出字段目标关联和模型输入字段默认值。
- 分流实验列表页和详情页的生产组、测试组条件分流。
- 操作数编译、模型运行时取值、规则输入依赖分析和血缘所需的引用信息。

下列内容不是操作数，不在本次范围：规则名称、显示标签、说明、权重、优先级、流量比例、初始分、测试参数表单，以及原始 QL 脚本编辑器的自由代码区域。

## 方案选择

### 方案一：继续扩展 VarPicker

在现有 VarPicker 中增加手输阈值和手输路径，并由每个页面继续维护自己的 `valueKind`、引用 ID 和依赖逻辑。该方案改动较少，但现有页面已经存在多套手动输入、变量选择和自动匹配实现，继续扩展会固化重复逻辑，函数参数等后续能力仍需再次分叉。

### 方案二：统一操作数模型与选择器

建立统一的操作数描述、选择器、表达式编译和依赖收集逻辑。各页面只声明当前位置允许的操作数类型，并接收统一结果。规则编译器和模型执行入口通过共享逻辑解析操作数。

这是本次采用的方案。它能满足当前范围，同时为函数、方法和复合表达式保留明确扩展点，不需要重写整个规则引擎。

### 方案三：完整表达式 AST 编辑器

将所有规则模型和编译器改写成通用表达式树。该方案扩展能力最强，但会同时重写九种设计器、动作系统和全部编译器，验证成本与当前需求不成比例。

## 统一操作数模型

操作数统一使用以下结构：

```json
{
  "kind": "LITERAL | PATH | REFERENCE | FUNCTION",
  "value": "100 或 request.customer.age",
  "valueType": "NUMBER",
  "refId": 123,
  "refType": "VARIABLE | CONSTANT | DATA_OBJECT | MODEL | MODEL_OUTPUT",
  "code": "request.customer.age",
  "label": "客户年龄",
  "functionId": null,
  "functionCode": null,
  "args": [],
  "resolved": true
}
```

字段含义：

- `LITERAL`：手输阈值。`value` 原样保存，`valueType` 决定编译时的引号和类型处理。
- `PATH`：手输路径。确认时自动查询统一引用目录；唯一匹配时补齐 ID 和引用类型，多项匹配时要求用户明确选择，未匹配时作为外部输入路径。
- `REFERENCE`：从变量、常量、数据对象或模型分类中明确选择的受管理引用，必须保存 `refId + refType`。
- `FUNCTION`：函数或方法调用，必须保存函数 ID，参数递归使用统一操作数。

`value`、`code` 和 `label` 保留用户输入及展示快照，但受管理引用的真实关联只认 `refId + refType`。重新加载时通过 ID 同步最新编码和标签。

## PATH 自动解析

手输路径确认时按以下规则处理：

1. 仅去除首尾空白，不改变大小写、驼峰或下划线。
2. 在当前项目和全局作用域的统一引用目录中，按完整 `scriptName`、完整对象字段路径和完整模型输出路径进行精确匹配。
3. 唯一命中时保留 `kind=PATH`，同时写入命中资源的 `refId`、`refType`、`valueType`、`code` 和 `label`，并标记 `resolved=true`。
4. 多项命中时在面板中展示候选项，用户明确选择前不能确认。
5. 未命中时保留用户原始路径、设置 `resolved=false`，将其作为外部运行时输入依赖。
6. 已解析路径再次加载时优先按 ID 恢复；资源编码变化后同步最新路径，不通过旧名称反向猜测。

这样，PATH 既能支持业务输入中的自由嵌套路径，也能在对应引擎变量、数据对象字段或模型输出时遵守 ID 关联约束。

## 选择器交互

新增统一 `OperandPicker`。外部控件是只读摘要触发器，点击后打开面板，不在获得焦点时自动弹出，也不在控件后方展示模式切换按钮。

面板左侧分类顺序固定为：

1. 手动输入
2. 变量
3. 常量
4. 数据对象
5. 模型
6. 函数或方法

“手动输入”分类右侧先展示“手输阈值”和“手输路径”两个入口：

- 手输阈值根据上下文类型显示数字、布尔、日期、枚举或普通文本编辑器。
- 手输路径显示路径输入框、自动解析状态和匹配候选；确认后返回 PATH 操作数。

其他分类复用现有字段目录、搜索、对象/模型分组、分页和已选字段能力。选择完成后，触发器显示类型、业务标签和编码摘要。

选择器通过 `allowedKinds` 和 `writableOnly` 控制上下文能力：

- 条件左值：PATH、可读取引用、函数；不允许 LITERAL。
- 条件右值：LITERAL、PATH、可读取引用、函数。
- 赋值目标和模型字段目标：PATH、可写变量、可写数据对象字段或模型输出；不允许 LITERAL 和常量。
- 模型字段默认值：LITERAL、PATH、可读取引用、函数。

新增类型只需注册分类、编辑器、摘要渲染、编译器和依赖收集器，不修改各业务页面的模式切换代码。

## 条件树

条件叶节点使用 `leftOperand`、`operator`、`rightOperand`：

```json
{
  "type": "leaf",
  "leftOperand": { "kind": "REFERENCE", "refId": 1, "refType": "VARIABLE" },
  "operator": ">=",
  "rightOperand": { "kind": "LITERAL", "value": "18", "valueType": "NUMBER" }
}
```

无右值运算符不显示右操作数。删除 `valueKind` 常量/变量下拉、右值输入框旁的字段切换按钮、`switchRightToVar` 和切换后自动打开选择器的逻辑。

决策表、规则集、决策树、决策流和分流实验全部使用同一个条件编辑器及同一操作数编译逻辑。

## 规则设计器与动作

各设计器把原来的 `varCode`、条件值、结果值和动作参数入口替换为操作数选择器。保存模型时持久化操作数对象；脚本生成时由共享操作数编译器转换为 QLExpress 表达式。

动作块按位置声明能力：赋值目标使用可写操作数，赋值值和函数参数使用值操作数。函数调用本身仍由动作块决定，函数参数使用递归操作数，从而避免当前 `wrapValue` 通过字符串形态猜测路径或常量。

## 模型字段

模型输入字段新增两个明确语义：

- `sourceOperand`：该模型特征从哪里取值。
- `defaultOperand`：来源不存在、为 null 或为空时使用的默认操作数。

模型输出字段使用 `targetOperand` 表示结果写入位置。

数据库为模型字段保存操作数 JSON，同时保留根引用的 `var_id`、`ref_type` 和 `script_name` 作为索引与查询投影；它们由服务端根据操作数统一同步，不再由页面分别维护。

模型执行流程：

1. 从当前执行上下文解析 `sourceOperand`。
2. 来源不存在、为 null 或为空时解析 `defaultOperand`。
3. 将解析结果转换为模型字段声明类型。
4. 以模型原始 `fieldName` 作为 PMML 入参键执行模型。

默认操作数为 PATH、REFERENCE 或 FUNCTION 时，同样按运行时语义求值，不作为普通字符串。

## 依赖分析

新增共享操作数依赖收集器，并递归处理函数参数：

- 已解析 PATH 和 REFERENCE：按 `refId + refType` 建立依赖，并继续使用现有变量来源、数据对象和模型输入展开逻辑。
- 未解析 PATH：按用户原始路径建立外部输入依赖字段。
- LITERAL：不产生依赖。
- FUNCTION：记录函数引用，并递归收集每个参数的依赖。

规则引用模型或模型输出时，模型的 `sourceOperand` 与 `defaultOperand` 依赖均展开到规则输入依赖字段中。默认路径因此会出现在规则测试 Schema、执行参数绑定和血缘分析中。

## 编译和运行时

前端提供共享操作数序列化、摘要和条件表达式辅助函数；后端核心模块提供统一操作数编译器，所有规则编译器调用同一入口：

- LITERAL 按 `valueType` 生成合法字面量。
- PATH 使用已解析资源的当前脚本名；未解析路径原样作为运行时输入路径。
- REFERENCE 通过 `refId + refType` 和 VarContext 取得脚本名。
- FUNCTION 通过函数 ID 取得函数编码并递归编译参数。

服务端提供统一运行时操作数解析器，支持嵌套 Map 路径和扁平完整路径键，并在模型执行前完成默认值解析和类型转换。

## 错误处理

- PATH 多项匹配时禁止确认并展示候选资源。
- 受管理引用缺少 ID、资源已删除或类型不兼容时，保存或发布校验返回具体字段位置和原因。
- 未解析 PATH 允许保存，但测试和执行缺少该路径时按现有缺失输入机制报告。
- 函数参数不完整时禁止保存对应规则或模型字段。
- 操作数解析不得修改用户输入的路径、编码或大小写。

## 测试与验收

前端测试覆盖：

- 分类顺序、上下文过滤、手输阈值和手输路径。
- PATH 唯一匹配、多项匹配、未匹配和按 ID 回显。
- 条件树没有常量/变量下拉、没有切换按钮、不会自动弹层。
- 各规则设计器、模型详情和实验页面正确保存统一操作数。

后端和核心测试覆盖：

- 四种操作数的编译与依赖收集。
- 条件树和动作表达式生成。
- 模型来源及默认操作数运行时解析、类型转换和缺失处理。
- 默认 PATH/REFERENCE/FUNCTION 依赖展开到规则输入依赖。
- 九种规则模型的现有编译和执行回归。

完成后执行仓库规定的完整前端与后端构建、启动和测试流程。最后通过浏览器从 UI 创建或编辑规则设计器、模型详情和分流实验条件，逐项选择阈值、路径和受管理引用，保存、重新打开并执行测试，确认交互、持久化、依赖和运行结果一致。
